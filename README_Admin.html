JWT Bearer Flow — Admin Setup

Step-by-step guide for Salesforce administrators to generate the private key and certificate, configure an External Client App (ECA), and provide the client with everything needed to run the JWT Bearer Token flow.

🗺️ Overview

This guide covers three phases of setup. All three must be completed before handing credentials to the client.

PhaseWhoWhat
Phase 1SF AdminGenerate certificate in Salesforce Certificate & Key Management, then export private key
Phase 2SF AdminCreate & configure External Client App (ECA) in Salesforce Setup
Phase 3SF AdminCollect & securely deliver credentials to the client
Phase 1

🔐 Generate Certificate & Private Key

✅ Primary Method Salesforce Certificate and Key Management

This is the security team's approved approach. All key generation happens inside Salesforce — no local certificate tooling required beyond keytool and openssl for the export conversion.

1
Create a Self-Signed Certificate in Salesforce
Navigate to:
Setup › Security › Certificate and Key Management › Create Self-Signed Certificate
  • ✓ Label: descriptive name (e.g. JWT_Auth_Cert)
  • ✓ Exportable Private Key: checked
  • ✓ Key Size: 2048

Click Save.

2
Export the Keystore (.jks)
From the Certificate and Key Management list, click Export to Keystore next to your certificate. Set a keystore password when prompted and save it securely. This downloads a .jks file to your machine.
3
Find the alias in the keystore
Terminal / PowerShellBASH
keytool -list -keystore <certname>.jks -storepass <jks_password>
ℹ️

keytool requires the JDK (not just the JRE). If not found, run: choco install openjdk -y then refreshenv.

4
Convert JKS → PKCS12 (.p12)
Terminal / PowerShellBASH
keytool -importkeystore \
  -srckeystore <certname>.jks \
  -destkeystore <certname>.p12 \
  -deststoretype PKCS12 \
  -srcalias <alias> \
  -srcstorepass <jks_password> \
  -deststorepass <p12_password> \
  -noprompt
5
Convert PKCS12 → PEM (private.key)
This produces private.key — the file you will deliver to the client.
Terminal / PowerShellBASH
openssl pkcs12 \
  -in <certname>.p12 \
  -nocerts \
  -nodes \
  -out private.key \
  -passin pass:<p12_password>
6
Verify the key
Terminal / PowerShellBASH
openssl rsa -in private.key -check -noout
# Expected: RSA key ok
7
Remove the intermediate .p12 file
Terminal / PowerShellBASH
# Mac / Linux
rm <certname>.p12

# Windows PowerShell
Remove-Item <certname>.p12

Files produced after Phase 1 (Method 1):

🗄️ <certname>.jks — Keystore · store securely
🔑 private.key — Private key · deliver to client
🚨

Never share private.key over email or Slack. Deliver it via a secure channel (1Password, Vault, encrypted ZIP, etc.). Anyone with this file can impersonate the integration user.

Alternative Method Generate via OpenSSL

Use this method only if Salesforce Certificate and Key Management is unavailable, or if your security team explicitly approves a locally-generated certificate.

1
Generate a password-protected RSA key
Terminal / PowerShellBASH
openssl genrsa -des3 -passout pass:x -out server.pass.key 2048
2
Strip the passphrase — generate the final private key
This produces server.key — the file you will give to the client.
Terminal / PowerShellBASH
openssl rsa -passin pass:x -in server.pass.key -out server.key
3
Remove the intermediate key file
Terminal / PowerShellBASH
# Mac / Linux
rm server.pass.key

# Windows PowerShell
Remove-Item server.pass.key
4
Create a Certificate Signing Request (CSR)
Fill in your organisation details when prompted (Country, State, Organisation, etc.).
Terminal / PowerShellBASH
openssl req -new -key server.key -out server.csr
5
Generate a self-signed certificate (valid 365 days)
This produces server.crt — the certificate you will upload to Salesforce ECA.
Terminal / PowerShellBASH
openssl x509 -req -sha256 -days 365 -in server.csr -signkey server.key -out server.crt

Files produced (Method 2):

🔑 server.key — Private key · deliver to client
📜 server.crt — Certificate · upload to Salesforce
📄 server.csr — CSR · can be kept or discarded
Phase 2

☁️ Configure External Client App (ECA) in Salesforce

Log in to Salesforce as an Administrator and follow these steps. Repeat for each org (Sandbox and/or Production).

1
Open App Manager
Setup › Apps › App Manager › New External Client App
2
Fill in the app details
Provide a descriptive name (e.g. Integration API — JWT), contact email, and a placeholder callback URL (e.g. https://localhost/callback).
3
Enable OAuth and select JWT Bearer Flow
Under the OAuth Settings section:
  • ✓ Check Enable OAuth Settings
  • ✓ Under Flow Enablement, select Enable JWT Bearer Flow
  • ✓ Upload server.crt in the Digital Signatures section
  • ✓ Add required OAuth scopes (e.g. api, web, refresh_token)
4
Save and copy the Consumer Key
After saving, navigate to the app's Settings tab and click Consumer Key and Secret. Copy the Consumer Key — this is the client_id / iss value for the client's jwt.properties.
ℹ️

Only the Consumer Key is needed for JWT Bearer flow — no client secret is required or transmitted.

5
Set OAuth policies and assign the integration user
App Manager › Your ECA › Manage › Edit Policies
  • ✓ Set Permitted Users to Admin approved users are pre-authorized
  • ✓ Under Profiles or Permission Sets, add the integration user's profile or permission set
6
Confirm your org's My Domain URL
Setup › My Domain
Copy the Current My Domain URL — this is the audience value for the client's jwt.properties.
Org TypeAudience Format
Sandboxhttps://[MyDomain]--[SandboxName].sandbox.my.salesforce.com
Productionhttps://[MyDomain].my.salesforce.com
Phase 3

📦 Deliver Credentials to Client

Collect the four values below and deliver them securely to the client so they can populate their jwt.properties file.

consumer_key
ECA Consumer Key — copied from Setup → App Manager → Your ECA → Settings tab
Where: Salesforce Setup → App Manager
username
Salesforce username of the integration user (the account assigned to the ECA policy)
Where: Setup → Users
server.key
The private key file generated in Phase 1 — deliver via a secure channel only
⚠️ Never send over email or Slack
audience
Your org's My Domain URL — found in Setup → My Domain
Format: https://[domain].my.salesforce.com
⚠️

The client places all four values in jwt.properties and runs java GenerateJWT --config jwt.properties. Refer them to README.html for full client instructions.

3 Configure OAuth Policy ⚠️ CRITICAL

⚠️

Without this step, all requests will fail with user hasn't approved this consumer.

  1. Go to the ECA detail page → Edit
  2. OAuth Policies → Permitted Users: Set to "Admin approved users are pre-authorized"
  3. Profiles → Manage Profiles → Add: System Administrator, SysAdmin_API, and any other profiles needing JWT access
  4. Click Save

Verify profile assignment:

SOQL VerificationSQL
SELECT Parent.Profile.Name, SetupEntityType
FROM SetupEntityAccess
WHERE SetupEntityType = 'ConnectedApplication'
  AND SetupEntityId = '<ECA_CONNECTED_APP_ID>'

If this returns 0 records, profiles are NOT assigned — redo this step.

4 Convert Certificate to Private Key (PEM)

4a — Find the aliasbash
keytool -list -keystore JWT_PeopleSoft.jks -storepass YOUR_PASSWORD
4b — Convert JKS → PKCS12bash
keytool -importkeystore \
  -srckeystore JWT_PeopleSoft.jks \
  -destkeystore JWT_PeopleSoft.p12 \
  -deststoretype PKCS12 \
  -srcalias <ALIAS_FROM_4a> \
  -srcstorepass YOUR_JKS_PASSWORD \
  -deststorepass YOUR_P12_PASSWORD
4c — Extract private keybash
openssl pkcs12 -in JWT_PeopleSoft.p12 -nocerts -nodes \
  -out private.key -passin pass:YOUR_P12_PASSWORD
4d — Extract public cert (for jwt.io testing)bash
openssl pkcs12 -in JWT_PeopleSoft.p12 -clcerts -nokeys \
  -out public.crt -passin pass:YOUR_P12_PASSWORD

🔄 Certificate Renewal

The self-signed certificate is valid for 365 days. Before expiry, follow the steps for whichever method was used to generate the original certificate.

ℹ️

Set a calendar reminder 30 days before expiry. A lapsed certificate will break the integration silently — the token exchange will return an invalid_grant error.

MethodRenewal Steps
Method 1 (Salesforce) Create a new cert in Certificate & Key Management → re-export JKS → repeat keytool/openssl conversion → update ECA Digital Signatures → re-deliver private.key
Method 2 (OpenSSL) Repeat Phase 1 OpenSSL steps → re-upload new server.crt to ECA → re-deliver new server.key
ActionFrequency
Renew certificate + re-upload to ECAAnnually (before certificate expiry)
Re-deliver private key to clientOnly when certificate is renewed
Consumer Key rotationOnly if ECA is recreated

📋 Notes

  • Method 1 requires the JDK (for keytool) and OpenSSL — install JDK via choco install openjdk -y on Windows; OpenSSL is available by default on Mac/Linux or via Win32 OpenSSL on Windows
  • Method 2 (OpenSSL) requires only OpenSSL — use only when approved by the security team as the alternative approach
  • The exported private.key is in PEM format, which the client's GenerateJWT.java loads directly with no conversion needed
  • Only the certificate (not the private key) goes into Salesforce — Salesforce uses it to verify the JWT signature without ever seeing the private key
  • Repeat Phase 2 for each org environment (Sandbox and Production) — each org gets its own ECA and Consumer Key
  • One certificate key pair can be reused across Sandbox and Production ECAs if preferred, or separate pairs can be generated per org
