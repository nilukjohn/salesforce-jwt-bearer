JWT Bearer Flow — Admin Setup

Step-by-step guide for Salesforce administrators to generate the private key and certificate, configure an External Client App (ECA), and provide the client with everything needed to run the JWT Bearer Token flow.

🗺️ Overview

This guide covers three phases of setup. All three must be completed before handing credentials to the client.

PhaseWhoWhat
Phase 1SF Admin / DevOpsGenerate private key & self-signed certificate using OpenSSL
Phase 2SF AdminCreate & configure External Client App (ECA) in Salesforce Setup
Phase 3SF AdminCollect & securely deliver credentials to the client
Phase 1

🔐 Generate Private Key & Certificate

Use OpenSSL to generate an RSA private key and a self-signed X.509 certificate. Run these commands in a terminal (Mac/Linux) or PowerShell with OpenSSL installed (Windows).

1
Generate a password-protected RSA key
Creates an intermediate passphrase-protected key (required by OpenSSL).
Terminal / PowerShellBASH
openssl genrsa -des3 -passout pass:x -out server.pass.key 2048
2
Strip the passphrase — generate the final private key
This produces server.key — the file you will give to the client.
Terminal / PowerShellBASH
openssl rsa -passin pass:x -in server.pass.key -out server.key
3
Remove the intermediate key file
Terminal / PowerShellBASH
# Mac / Linux
rm server.pass.key

# Windows PowerShell
Remove-Item server.pass.key
4
Create a Certificate Signing Request (CSR)
Fill in your organisation details when prompted (Country, State, Organisation, etc.).
Terminal / PowerShellBASH
openssl req -new -key server.key -out server.csr
5
Generate a self-signed certificate (valid 365 days)
This produces server.crt — the certificate you will upload to Salesforce.
Terminal / PowerShellBASH
openssl x509 -req -sha256 -days 365 -in server.csr -signkey server.key -out server.crt

Files produced after Phase 1:

🔑 server.key — Private key · deliver to client
📜 server.crt — Certificate · upload to Salesforce
📄 server.csr — CSR · can be kept or discarded
🚨

Never share server.key over email or Slack. Deliver it via a secure channel (1Password, Vault, encrypted ZIP, etc.). Anyone with this file can impersonate the integration user.

Phase 2

☁️ Configure External Client App (ECA) in Salesforce

Log in to Salesforce as an Administrator and follow these steps. Repeat for each org (Sandbox and/or Production).

1
Open App Manager
Setup › Apps › App Manager › New External Client App
2
Fill in the app details
Provide a descriptive name (e.g. Integration API — JWT), contact email, and a placeholder callback URL (e.g. https://localhost/callback).
3
Enable OAuth and select JWT Bearer Flow
Under the OAuth Settings section:
  • ✓ Check Enable OAuth Settings
  • ✓ Under Flow Enablement, select Enable JWT Bearer Flow
  • ✓ Upload server.crt in the Digital Signatures section
  • ✓ Add required OAuth scopes (e.g. api, web, refresh_token)
4
Save and copy the Consumer Key
After saving, navigate to the app's Settings tab and click Consumer Key and Secret. Copy the Consumer Key — this is the client_id / iss value for the client's jwt.properties.
ℹ️

Only the Consumer Key is needed for JWT Bearer flow — no client secret is required or transmitted.

5
Set OAuth policies and assign the integration user
App Manager › Your ECA › Manage › Edit Policies
  • ✓ Set Permitted Users to Admin approved users are pre-authorized
  • ✓ Under Profiles or Permission Sets, add the integration user's profile or permission set
6
Confirm your org's My Domain URL
Setup › My Domain
Copy the Current My Domain URL — this is the audience value for the client's jwt.properties.
Org TypeAudience Format
Sandboxhttps://[MyDomain]--[SandboxName].sandbox.my.salesforce.com
Productionhttps://[MyDomain].my.salesforce.com
Phase 3

📦 Deliver Credentials to Client

Collect the four values below and deliver them securely to the client so they can populate their jwt.properties file.

consumer_key
ECA Consumer Key — copied from Setup → App Manager → Your ECA → Settings tab
Where: Salesforce Setup → App Manager
username
Salesforce username of the integration user (the account assigned to the ECA policy)
Where: Setup → Users
server.key
The private key file generated in Phase 1 — deliver via a secure channel only
⚠️ Never send over email or Slack
audience
Your org's My Domain URL — found in Setup → My Domain
Format: https://[domain].my.salesforce.com
⚠️

The client places all four values in jwt.properties and runs java GenerateJWT --config jwt.properties. Refer them to README.html for full client instructions.

🔄 Certificate Renewal

The self-signed certificate is valid for 365 days. Before expiry, repeat Phase 1 to generate a new key pair, re-upload the new server.crt to the ECA, and re-deliver the new server.key to the client.

ℹ️

Set a calendar reminder 30 days before expiry. A lapsed certificate will break the integration silently — the token exchange will return an invalid_grant error.

ActionFrequency
Renew certificate + re-upload to ECAAnnually (before server.crt expiry)
Re-deliver server.key to clientOnly when certificate is renewed
Consumer Key rotationOnly if ECA is recreated

📋 Notes

  • OpenSSL must be installed locally to run Phase 1 commands — available on Mac/Linux by default; install via Win32 OpenSSL on Windows
  • The generated server.key is in PKCS#8 PEM format, which the client's GenerateJWT.java loads directly with no conversion
  • Only server.crt goes into Salesforce — Salesforce uses it to verify the JWT signature without ever seeing the private key
  • Repeat Phase 2 for each org environment (Sandbox and Production) — each org gets its own ECA and Consumer Key
  • One certificate key pair can be reused across Sandbox and Production ECAs if preferred, or separate pairs can be generated per org
